
[size=20][b]Обзор[/b][/size]

Система GameData позволяет плагинам получать доступ к внутренним компонентам игры через [b]сигнатуры[/b], [b]смещения[/b], [b]адреса[/b], [b]патчи[/b] и [b]виртуальные таблицы[/b]. Это позволяет перехватывать функции, читать память и изменять поведение игры без жёстко закодированных адресов.

Файлы GameData - это конфигурационные файлы на основе JSON, которые содержат платформозависимые шаблоны и смещения. Система автоматически обрабатывает различия платформ (Windows/Linux) и обновления игры, используя сканирование сигнатур вместо жёстко закодированных адресов.

[quote][b]Info:[/b] - Платформонезависимый код (Windows/Linux обрабатываются автоматически)
[list]
[*]Устойчивость к обновлениям игры (сигнатуры автоматически находят адреса)
[*]Централизованная конфигурация для операций с памятью
[*]Автоматическая поддержка бинарных патчей[/quote]
[/list]

[size=20][b]Подходы к API[/b][/size]

s2sdk предоставляет два способа работы с gamedata:

[list=1]
[*][b]Функциональный API[/b] - Ручное управление дескрипторами с помощью [font=monospace]LoadGameConfigFile[/font] / [font=monospace]CloseGameConfigFile[/font]
[*][b]Объектно-ориентированный API[/b] - RAII-обёртка с использованием класса [font=monospace]GameConfig[/font] (автоматическая очистка)
[/list]

[quote][b]Info:[/b] [b]Рекомендуется:[/b] Используйте [b]класс GameConfig[/b] для автоматического управления ресурсами (RAII). Он автоматически закрывает файл gamedata при уничтожении объекта, предотвращая утечки памяти.[/quote]

[size=20][b]Загрузка и закрытие файлов GameData (функциональный API)[/b][/size]

Используйте [font=monospace]LoadGameConfigFile[/font] для загрузки пользовательского файла gamedata и [font=monospace]CloseGameConfigFile[/font] по завершении.

[quote][b]Info:[/b] [b]Массив путей:[/b] [font=monospace]LoadGameConfigFile[/font] принимает [b]массив путей к файлам[/b] для поиска файла gamedata. Чтобы загрузить из директории вашего плагина, постройте полный путь, используя расположение вашего плагина.[/quote]

        // Use gamedata methods...
        int offset = GetGameConfigOffset(gameData, "SomeOffset");
        PrintToServer($"Offset value: {offset}\n");

    {
        // Clean up gamedata
        if (gameData != IntPtr.Zero)
        {
            CloseGameConfigFile(gameData);
            gameData = IntPtr.Zero;
        }
    }

[size=20][b]Использование класса GameConfig (объектно-ориентированный API)[/b][/size]

Класс [font=monospace]GameConfig[/font] - это [b]RAII-обёртка[/b], которая автоматически управляет ресурсами gamedata. Он создаётся с помощью [font=monospace]LoadGameConfigFile[/font] и автоматически вызывает [font=monospace]CloseGameConfigFile[/font] при уничтожении.

[quote][b]Info:[/b] - Автоматическая очистка ресурсов (паттерн RAII)
[list]
[*]Не требуется ручной вызов [font=monospace]CloseGameConfigFile[/font]
[*]Более чистый и поддерживаемый код
[*]Типобезопасный интерфейс с методами-членами[/quote]
[/list]

[size=16][b]Методы GameConfig[/b][/size]

Класс [font=monospace]GameConfig[/font] предоставляет следующие методы:
[list]
[*][font=monospace]GetPatch(name)[/font] - Получить адрес, по которому был применён патч
[*][font=monospace]GetOffset(name)[/font] - Получить значение именованного смещения
[*][font=monospace]GetAddress(name)[/font] - Получить разрешённый адрес
[*][font=monospace]GetVTable(name)[/font] - Получить адрес виртуальной таблицы
[*][font=monospace]GetSignature(name)[/font] - Получить адрес именованной сигнатуры
[/list]

[size=20][b]Сигнатуры[/b][/size]

Сигнатуры - это байтовые шаблоны, используемые для поиска функций или данных в игровых бинарных файлах. Они устойчивы к незначительным обновлениям игры.

[size=16][b]Формат сигнатуры[/b][/size]

Сигнатуры используют шестнадцатеричные байты с [font=monospace]?[/font] в качестве подстановочных знаков:
[list]
[*][font=monospace]48 89 5C 24[/font] - Точные байты
[*][font=monospace]?[/font] или [font=monospace]??[/font] - Любой одиночный байт (подстановочный знак)
[*]Шестнадцатеричные значения, разделённые пробелами
[/list]

[size=16][b]Генерация сигнатур[/b][/size]

Вы можете генерировать сигнатуры с помощью:

[list=1]
[*][b]IDA Pro[/b] с плагином [url=https://github.com/ajkhoury/SigMaker-x64]sigmaker[/url]
[*][b]Binary Ninja[/b] со скриптами генерации сигнатур
[*][b]Ghidra[/b] с пользовательскими скриптами
[/list]

[quote][b]Info:[/b] [b]Именование сигнатур:[/b] Если сигнатура начинается с [font=monospace]@[/font] (например, [font=monospace]@FunctionName[/font]), система будет обрабатывать её как [b]имя символа[/b] вместо байтового шаблона. Это полезно для экспортируемых функций.[/quote]

[size=20][b]Адреса[/b][/size]

Адреса объединяют сигнатуры с вычислениями смещений для определения конкретных позиций в памяти. Они поддерживают три типа разрешения:

[size=16][b]Типы адресов[/b][/size]

[list=1]
[*][b][font=monospace]offset[/font][/b] - Добавить смещение к адресу сигнатуры
[*][b][font=monospace]read[/font][/b] - Прочитать указатель по адресу сигнатуры + смещение, следовать N раз
[*][b][font=monospace]read_offs32[/font][/b] - Прочитать 32-битное относительное смещение, вычислить абсолютный адрес
[/list]

[size=16][b]Определение адресов[/b][/size]

Адреса определяются как [b]массивы шагов действий[/b] для каждой платформы:
[code=json]
{
  "Addresses": {
    "GlobalVars": {
      "signature": "GlobalVarsSignature",
      "win64": [
        {
          "offset": 3
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 3
        }
      ]
    },
    "EntityList": {
      "signature": "EntityListSignature",
      "win64": [
        {
          "offset": 10
        },
        {
          "read": 2
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 8
        },
        {
          "read": 2
        }
      ]
    },
    "SomeFunction": {
      "signature": "FunctionCallSignature",
      "win64": [
        {
          "read_offs32": 1
        }
      ],
      "linuxsteamrt64": [
        {
          "read_offs32": 1
        }
      ]
    }
  }
}
[/code]

[quote][b]Info:[/b] [b]Выполнение шагов:[/b] Каждое действие адреса выполняется последовательно. Например, [font=monospace]EntityList[/font] сначала добавляет смещение 10, затем читает указатель 2 раза.[/quote]

[size=20][b]Патчи[/b][/size]

Патчи позволяют автоматически изменять игровые бинарные файлы во время выполнения. Они находят код с помощью сигнатур и заменяют байты.

[quote][b]Info:[/b] [b]Предупреждение:[/b] Патчи напрямую изменяют память игры. Неправильные патчи могут привести к сбою сервера. Всегда проверяйте байты перед применением патчей![/quote]

[size=16][b]Определение патчей[/b][/size]

Патчи ссылаются на адрес (из разделов Addresses или Signatures) и указывают байты патча для записи:
[code=json]
{
  "Patches": {
    "VScriptEnable": {
      "address": "VScriptInitialization",
      "win64": "BE 02",
      "linuxsteamrt64": "83 FE 02"
    },
    "NoValidation": {
      "address": "ValidationCheck",
      "win64": "EB",
      "linuxsteamrt64": "90 90"
    }
  }
}
[/code]

[list]
[*][b][font=monospace]address[/font][/b] - Имя адреса/сигнатуры из файла gamedata
[*][b][font=monospace]win64[/font] / [font=monospace]linuxsteamrt64[/font][/b] - Шестнадцатеричные байты для записи (байты патча)
[/list]

[quote][b]Info:[/b] [b]Осторожно:[/b] Патчи напрямую изменяют память игры и применяются автоматически при загрузке. Тщательно тестируйте, чтобы избежать сбоев![/quote]

[quote][b]Info:[/b] [b]Автоматическое применение:[/b] Патчи применяются автоматически при загрузке файла gamedata. Вам не нужно вызывать какие-либо функции - система обрабатывает это![/quote]

| Метод | Описание |

| Метод | Описание |

[size=20][b]Советы и лучшие практики[/b][/size]

[list=1]
[*][b]Всегда закрывайте файлы gamedata[/b] - Используйте [font=monospace]CloseGameConfigFile()[/font] в [font=monospace]OnPluginEnd()[/font] для предотвращения утечек памяти
[*][b]Проверяйте сигнатуры[/b] - Проверяйте, что возвращённые адреса не равны null перед их использованием
[*][b]Используйте осмысленные имена[/b] - Давайте описательные имена вашим сигнатурам, смещениям и адресам
[*][b]Документируйте ваши шаблоны[/b] - Комментируйте, откуда взялись сигнатуры и на что они указывают
[*][b]Тестируйте на всех платформах[/b] - Убедитесь, что шаблоны для Windows и Linux верны
[*][b]Держите gamedata в актуальном состоянии[/b] - Обновляйте сигнатуры, когда обновления игры их ломают
[*][b]Используйте verify bytes в патчах[/b] - Всегда указывайте [font=monospace]verify[/font] для предотвращения неправильного применения патчей
[*][b]Кешируйте адреса[/b] - Сохраняйте разрешённые адреса вместо повторного вызова методов Get
[*][b]Организуйте ваш JSON[/b] - Группируйте связанные сигнатуры, смещения и адреса вместе
[*][b]Версионируйте ваш gamedata[/b] - Рассмотрите возможность добавления комментариев с версиями для отслеживания изменений
[/list]

[size=20][b]Устранение неполадок[/b][/size]

[size=16][b]Сигнатура не найдена[/b][/size]
[list]
[*][b]Причина[/b]: Шаблон не соответствует игровому бинарному файлу
[*][b]Решение[/b]: Перегенерируйте сигнатуру, используя IDA/Binary Ninja, проверьте платформозависимые шаблоны
[/list]

[size=16][b]Ошибка проверки патча[/b][/size]
[list]
[*][b]Причина[/b]: Игра была обновлена, и байты изменились
[*][b]Решение[/b]: Обновите байты [font=monospace]verify[/font] и [font=monospace]patch[/font] для новой версии игры
[/list]

[size=16][b]Возвращён недействительный адрес[/b][/size]
[list]
[*][b]Причина[/b]: Сигнатура не сработала или вычисление смещения неверно
[*][b]Решение[/b]: Проверьте работу сигнатуры, проверьте значения [font=monospace]offset[/font] и [font=monospace]read[/font]
[/list]

[size=16][b]Файл gamedata не загружается[/b][/size]
[list]
[*][b]Причина[/b]: Файл не найден или недействительный JSON
[*][b]Решение[/b]: Проверьте путь к файлу, валидируйте синтаксис JSON
[/list]

[quote][b]Info:[/b] [b]Различия платформ:[/b] Всегда тестируйте ваши gamedata на Windows и Linux. Сигнатуры и смещения часто различаются между платформами![/quote]

[size=20][b]Продвинутые темы[/b][/size]

[size=16][b]Сигнатуры на основе символов[/b][/size]

Для экспортируемых функций используйте префикс [font=monospace]@[/font] для ссылки по имени символа:
[code=json]
{
  "Signatures": {
    "CreateInterface": {
      "windows": {
        "library": "server",
        "signature": "@CreateInterface"
      },
      "linux": {
        "library": "server",
        "signature": "@CreateInterface"
      }
    }
  }
}
[/code]

[size=16][b]Многоуровневое разрешение адресов[/b][/size]

Комбинируйте [font=monospace]read[/font] со смещениями для следования цепочкам указателей:
[code=json]
{
  "Addresses": {
    "DeepPointer": {
      "signature": "BaseSignature",
      "win64": [
        {
          "offset": 10
        },
        {
          "read": 3
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 8
        },
        {
          "read": 3
        }
      ]
    }
  }
}
[/code]

Это сначала добавляет смещение, затем читает указатель 3 раза (следует 3 разыменованиям указателей).

[size=16][b]Адреса с относительным смещением (read_offs32)[/b][/size]

Используйте [font=monospace]read_offs32[/font] для RIP-относительной адресации (распространено в x64). Это читает 32-битное относительное смещение и вычисляет абсолютный адрес:
[code=json]
{
  "Addresses": {
    "v8::Isolate::Enter": {
      "signature": "CSScript::ResolveModule",
      "win64": [
        {
          "read_offs32": 59
        }
      ],
      "linuxsteamrt64": [
        {
          "read_offs32": 45
        }
      ]
    }
  }
}
[/code]

Это полезно, когда сигнатура указывает на инструкции вроде [font=monospace]call [rip + offset][/font] или [font=monospace]lea reg, [rip + offset][/font]. Значение [font=monospace]read_offs32[/font] - это байтовое смещение, где находится 32-битное относительное смещение.
