## Обзор

Система GameData позволяет плагинам получать доступ к внутренним компонентам игры через **сигнатуры**, **смещения**, **адреса**, **патчи** и **виртуальные таблицы**. Это позволяет перехватывать функции, читать память и изменять поведение игры без жестко закодированных адресов.

Файлы GameData — это конфигурационные файлы на основе JSON, которые содержат платформо-зависимые паттерны и смещения. Система автоматически обрабатывает различия между платформами (Windows/Linux) и обновления игры, используя сканирование сигнатур вместо жестко закодированных адресов.

::callout{type="info"}
**Ключевые преимущества:**
- Платформо-независимый код (Windows/Linux обрабатываются автоматически)
- Устойчивость к обновлениям игры (сигнатуры автоматически находят адреса)
- Централизованная конфигурация для операций с памятью
- Автоматическая поддержка бинарных патчей
::

## Подходы к API

s2sdk предоставляет два способа работы с gamedata:

1. **Функциональный API** - Ручное управление дескрипторами с помощью `LoadGameConfigFile` / `CloseGameConfigFile`
2. **Объектно-ориентированный API** - RAII-обертка с использованием класса `GameConfig` (автоматическая очистка)

::callout{type="info"}
**Рекомендуется:** Используйте **класс GameConfig** для автоматического управления ресурсами (RAII). Он автоматически закрывает файл gamedata при уничтожении объекта, предотвращая утечки памяти.
::

## Загрузка и закрытие файлов GameData (Функциональный API)

Используйте `LoadGameConfigFile` для загрузки пользовательского файла gamedata и `CloseGameConfigFile` после завершения работы.

::callout{type="info"}
**Массив путей:** `LoadGameConfigFile` принимает **массив путей к файлам** для поиска файла gamedata. Чтобы загрузить из директории вашего плагина, создайте полный путь, используя расположение вашего плагина.
::

### Получение расположения плагина

Каждый язык предоставляет способ доступа к директории плагина:

| Язык | Метод |
|----------|--------|
| C# | `this.GetLocation()` |
| C++ | `this->GetLocation()` |
| Python | `self.location` |
| Go | `plugify.Plugin.Location` |
| JavaScript | `this.location` |
| Lua | `self.location` |

using Plugify;
using static s2sdk.s2sdk;

public unsafe class Sample : Plugin
{
    private nint gameData;

    public void OnPluginStart()
    {
        // Создание пути к файлу gamedata в директории плагина

    {
        // Очистка gamedata
        if (gameData != IntPtr.Zero)
        {

        }
    }

## Структура JSON-схемы

Файлы GameData используют формат JSON с платформо-зависимыми конфигурациями. Вот базовая структура:

```json
{
  "Signatures": {
    "SignatureName": {
      "library": "server",
      "win64": "48 89 5C 24 ? 48 89 6C 24 ? 48 89 74 24 ? 57 41 56 41 57 48 83 EC 40",
      "linuxsteamrt64": "55 48 89 E5 41 57 41 56 41 55 41 54 53 48 83 EC 18"
    }
  },
  "Offsets": {
    "OffsetName": {
      "win64": 123,
      "linuxsteamrt64": 456
    }
  },
  "Addresses": {
    "AddressName": {
      "signature": "SignatureName",
      "win64": [
        {
          "offset": 10
        },
        {
          "read": 2
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 5
        },
        {
          "read": 2
        }
      ]
    }
  },
  "Patches": {
    "PatchName": {
      "address": "SignatureName",
      "win64": "EB",
      "linuxsteamrt64": "90 90"
    }
  },
  "VTables": {
    "VTableName": {
      "library": "server",
      "table": "CBaseEntity"
    }
  }
}
```

::callout{type="warning"}
**Важно:** Файл gamedata должен быть размещен в директории данных вашего плагина. Имя файла, передаваемое в `LoadGameConfigFile`, должно быть относительным к этой директории.
::

## Использование класса GameConfig (Объектно-ориентированный API)

Класс `GameConfig` — это **RAII-обертка**, которая автоматически управляет ресурсами gamedata. Он создается с использованием `LoadGameConfigFile` и автоматически вызывает `CloseGameConfigFile` при уничтожении.

::callout{type="success"}
**Преимущества класса GameConfig:**
- Автоматическая очистка ресурсов (паттерн RAII)
- Не требуется ручной вызов `CloseGameConfigFile`
- Более чистый, поддерживаемый код
- Типобезопасный интерфейс с методами класса
::

### Методы GameConfig

Класс `GameConfig` предоставляет следующие методы:
- `GetPatch(name)` - Получить адрес, где был применен патч
- `GetOffset(name)` - Получить именованное значение смещения
- `GetAddress(name)` - Получить разрешенный адрес
- `GetVTable(name)` - Получить адрес виртуальной таблицы
- `GetSignature(name)` - Получить адрес именованной сигнатуры

## Сигнатуры

Сигнатуры — это байтовые паттерны, используемые для поиска функций или данных в игровых бинарных файлах. Они устойчивы к незначительным обновлениям игры.

### Формат сигнатуры

Сигнатуры используют шестнадцатеричные байты с `?` в качестве подстановочных знаков:
- `48 89 5C 24` - Точные байты
- `?` или `??` - Любой одиночный байт (подстановочный знак)
- Шестнадцатеричные значения, разделенные пробелами

### Генерация сигнатур

Вы можете генерировать сигнатуры, используя:

1. **IDA Pro** с [плагином sigmaker](https://github.com/ajkhoury/SigMaker-x64)
2. **Binary Ninja** со скриптами генерации сигнатур
3. **Ghidra** с пользовательскими скриптами

::callout{type="info"}
**Именование сигнатур:** Если сигнатура начинается с `@` (например, `@FunctionName`), система будет рассматривать её как **имя символа** вместо байтового паттерна. Это полезно для экспортированных функций.
::

## Адреса

Адреса комбинируют сигнатуры с расчетами смещений для определения конкретных позиций в памяти. Они поддерживают три типа разрешения:

### Типы адресов

1. **`offset`** - Добавить смещение к адресу сигнатуры
2. **`read`** - Прочитать указатель по адресу сигнатура + смещение, следовать N раз
3. **`read_offs32`** - Прочитать 32-битное относительное смещение, вычислить абсолютный адрес

### Определение адресов

Адреса определяются как **массивы шагов действий** для каждой платформы:

```json
{
  "Addresses": {
    "GlobalVars": {
      "signature": "GlobalVarsSignature",
      "win64": [
        {
          "offset": 3
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 3
        }
      ]
    },
    "EntityList": {
      "signature": "EntityListSignature",
      "win64": [
        {
          "offset": 10
        },
        {
          "read": 2
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 8
        },
        {
          "read": 2
        }
      ]
    },
    "SomeFunction": {
      "signature": "FunctionCallSignature",
      "win64": [
        {
          "read_offs32": 1
        }
      ],
      "linuxsteamrt64": [
        {
          "read_offs32": 1
        }
      ]
    }
  }
}
```

::callout{type="info"}
**Выполнение шагов:** Каждое действие адреса выполняется последовательно. Например, `EntityList` сначала добавляет смещение 10, затем читает указатель 2 раза.

## Патчи

Патчи позволяют автоматически модифицировать игровые бинарные файлы во время выполнения. Они находят код с использованием сигнатур и заменяют байты.

::callout{type="danger"}
**Предупреждение:** Патчи напрямую модифицируют память игры. Неправильные патчи могут привести к сбою сервера. Всегда проверяйте байты перед применением патча!
::

### Определение патчей

Патчи ссылаются на адрес (из секций Addresses или Signatures) и указывают байты патча для записи:

```json
{
  "Patches": {
    "VScriptEnable": {
      "address": "VScriptInitialization",
      "win64": "BE 02",
      "linuxsteamrt64": "83 FE 02"
    },
    "NoValidation": {
      "address": "ValidationCheck",
      "win64": "EB",
      "linuxsteamrt64": "90 90"
    }
  }
}
```

- **`address`** - Имя адреса/сигнатуры из файла gamedata
- **`win64` / `linuxsteamrt64`** - Шестнадцатеричные байты для записи (байты патча)

::callout{type="warning"}
**Осторожно:** Патчи напрямую модифицируют память игры и автоматически применяются при загрузке. Тщательно тестируйте, чтобы избежать сбоев!
::

::callout{type="info"}
**Автоматическое применение:** Патчи автоматически применяются при загрузке файла gamedata. Вам не нужно вызывать какие-либо функции - система обрабатывает это!
::

## Справочник методов

### Управление файлами
| Метод | Описание |
|--------|-------------|
| `LoadGameConfigFile(filename)` | Загрузить JSON-файл gamedata. Возвращает дескриптор или null при ошибке. |
| `CloseGameConfigFile(handle)` | Закрыть и освободить файл gamedata. |

### Сигнатуры
| Метод | Описание |
|--------|-------------|
| `GetSignature(handle, name)` | Получить адрес именованной сигнатуры. |
| `GetMemSig(handle, name)` | Альтернативный метод для получения адреса сигнатуры. |
| `GetAllSignatures(handle)` | Получить массив всех имен сигнатур. |

### Смещения
| Метод | Описание |
|--------|-------------|
| `GetOffset(handle, name)` | Получить именованное значение смещения. |
| `GetAllOffsets(handle)` | Получить массив всех имен смещений. |

### Адреса
| Метод | Описание |
|--------|-------------|
| `GetAddress(handle, name)` | Получить разрешенный адрес (сигнатура + расчет смещения). |
| `GetAllAddresses(handle)` | Получить массив всех имен адресов. |

### Патчи
| Метод | Описание |
|--------|-------------|
| `GetPatch(handle, name)` | Получить адрес, где был применен патч. |
| `GetAllPatches(handle)` | Получить массив всех имен патчей. |

::callout{type="info"}
**Примечание:** Патчи автоматически применяются при загрузке gamedata. Используйте `GetPatch()` для проверки расположения патчей, а не для их применения.
::

### VTables
| Метод | Описание |
|--------|-------------|
| `GetVTable(handle, name)` | Получить адрес виртуальной таблицы. |
| `GetAllVTables(handle)` | Получить массив всех имен vtable. |

## Советы и лучшие практики

1. **Всегда закрывайте файлы gamedata** - Используйте `CloseGameConfigFile()` в `OnPluginEnd()` для предотвращения утечек памяти
2. **Проверяйте сигнатуры** - Проверьте, что возвращаемые адреса не являются null перед их использованием
3. **Используйте осмысленные имена** - Именуйте ваши сигнатуры, смещения и адреса описательно
4. **Документируйте паттерны** - Комментируйте, откуда взялись сигнатуры и на что они указывают
5. **Тестируйте на всех платформах** - Убедитесь, что паттерны для Windows и Linux корректны
6. **Обновляйте gamedata** - Обновляйте сигнатуры, когда обновления игры их ломают
7. **Используйте verify bytes в патчах** - Всегда указывайте `verify` для предотвращения неправильного применения патчей
8. **Кэшируйте адреса** - Сохраняйте разрешенные адреса вместо повторного вызова методов Get
9. **Организуйте ваш JSON** - Группируйте связанные сигнатуры, смещения и адреса вместе
10. **Версионируйте ваш gamedata** - Рассмотрите добавление комментариев версий для отслеживания изменений

## Устранение неполадок

### Сигнатура не найдена
- **Причина**: Паттерн не соответствует игровому бинарному файлу
- **Решение**: Пересоздайте сигнатуру с помощью IDA/Binary Ninja, проверьте платформо-зависимые паттерны

### Ошибка проверки патча
- **Причина**: Игра была обновлена и байты изменились
- **Решение**: Обновите байты `verify` и `patch` для новой версии игры

### Возвращен недействительный адрес
- **Причина**: Сигнатура не сработала или расчет смещения неверен
- **Решение**: Проверьте, работает ли сигнатура, проверьте значения `offset` и `read`

### Файл Gamedata не загружается
- **Причина**: Файл не найден или недействительный JSON
- **Решение**: Проверьте путь к файлу, проверьте синтаксис JSON

::callout{type="warning"}
**Различия платформ:** Всегда тестируйте ваш gamedata на Windows и Linux. Сигнатуры и смещения часто отличаются между платформами!
::

## Продвинутые темы

### Сигнатуры на основе символов

Для экспортированных функций используйте префикс `@` для ссылки по имени символа:

```json
{
  "Signatures": {
    "CreateInterface": {
      "windows": {
        "library": "server",
        "signature": "@CreateInterface"
      },
      "linux": {
        "library": "server",
        "signature": "@CreateInterface"
      }
    }
  }
}
```

### Многоуровневое разрешение адресов

Комбинируйте `read` со смещениями для следования цепочкам указателей:

```json
{
  "Addresses": {
    "DeepPointer": {
      "signature": "BaseSignature",
      "win64": [
        {
          "offset": 10
        },
        {
          "read": 3
        }
      ],
      "linuxsteamrt64": [
        {
          "offset": 8
        },
        {
          "read": 3
        }
      ]
    }
  }
}
```

Это сначала добавляет смещение, затем читает указатель 3 раза (следует 3 разыменованиям указателей).

### Адреса с относительным смещением (read_offs32)

Используйте `read_offs32` для RIP-относительной адресации (распространено в x64). Это читает 32-битное относительное смещение и вычисляет абсолютный адрес:

```json
{
  "Addresses": {
    "v8::Isolate::Enter": {
      "signature": "CSScript::ResolveModule",
      "win64": [
        {
          "read_offs32": 59
        }
      ],
      "linuxsteamrt64": [
        {
          "read_offs32": 45
        }
      ]
    }
  }
}
```

Это полезно, когда сигнатура указывает на инструкции типа `call [rip + offset]` или `lea reg, [rip + offset]`. Значение `read_offs32` — это байтовое смещение, где находится 32-битное относительное смещение.
